# Fossil Server Cluster — Design Spec

**Date**: 2026-05-14
**Status**: Approved for plan
**Author**: tmk + Bot (brainstorming session)

## 1. Purpose & Scope

Design a small, hardened NixOS-based fossil-server cluster for self-hosting a personal project. Three servers across two cloud providers (two on Hetzner in different data centres, one on DigitalOcean added later), each running a full read/write fossil server. The canonical server is authoritative for users; the two secondaries automatically stay in sync via periodic `fossil all sync -u` against the canonical, following the [canonical fossil-scm.org self-hosting pattern](https://fossil-scm.org/home/doc/trunk/www/selfhost.wiki).

**In scope**

- NixOS module that runs fossil with native TLS (no reverse proxy).
- ACME via NixOS `security.acme` with Cloudflare DNS-01.
- Per-host flake outputs deployable via `nixos-anywhere`.
- agenix-managed secrets in the repo.
- Tailscale on every host alongside public SSH.
- Auto-upgrade weekly with reboot.
- Comprehensive documentation set (architecture, setup, operations, runbooks, reference).

**Explicit non-goals for v1**

- Offsite backups beyond replication (replication ≠ backup, gap documented; revisit later).
- Email alerts for arbitrary unit failures (only the sync timer is monitored, via healthchecks.io).
- Automatic failover/promotion of a secondary (manual runbook only).
- Web UI for repo management (repo creation is a scripted action).
- NixOS VM integration tests (covered by `nix flake check` + per-target eval + post-deploy smoke test).
- Fossil git-export mirror.

## 2. Topology & Repo Layout

### Cluster topology

| Role        | Hostname (DNS)                | Hardware                          | Sync timer                  |
| ----------- | ----------------------------- | --------------------------------- | --------------------------- |
| canonical   | `fossil.exidia.com`           | Hetzner Cloud, DC1 (e.g. Nuremberg), CPX11 | — (serves users only)     |
| secondary-1 | `s1.fossil.exidia.com`        | Hetzner Cloud, DC2 (e.g. Falkenstein), CPX11 | `fossil all sync -u` every 5 min |
| secondary-2 | `s2.fossil.exidia.com`        | DigitalOcean, region selected at provisioning (added later) | `fossil all sync -u` every 5 min |

Convergence — explicit timings with a 5-minute sync interval:

- Write originated on **canonical** → reaches each secondary within ~5 min worst-case (one timer tick).
- Write originated on **a secondary** → reaches canonical within ~5 min (its own next tick pushes), then reaches the *other* secondary within ~5 min more (the other's next tick pulls). **Worst-case secondary→secondary convergence: ~10 min.**

All three hosts accept user writes; fossil's append-only artifact model means concurrent edits don't destroy data, they create independent branch leaves that can be merged later.

### DNS

Subdomain `fossil.exidia.com` is delegated from Hover (registrar) to Cloudflare (DNS hosting). The rest of `exidia.com` stays on Hover. Cloudflare zone holds:

- `@` (apex) → canonical IP
- `s1` → secondary-1 IP
- `s2` → secondary-2 IP (added with secondary-2)

### Per-host filesystem layout

- `/var/lib/fossil/museum/` — repolist directory, holds N `.fossil` files. Owned by the `fossil` system user (mode `0750`).
- `/var/lib/acme/<domain>/` — `security.acme`-managed cert files; group-readable by `fossil`.

### Repo URLs

Repos served as `https://<host>/<reponame>` via `fossil server --repolist /var/lib/fossil/museum`. E.g.:

- `https://fossil.exidia.com/code`
- `https://fossil.exidia.com/notes`
- `https://s1.fossil.exidia.com/code` (mirror of the same repo)

## 3. NixOS Repo Structure

Fresh repo at `~/dev/my/nixos-fossil`, hosted publicly at `github:teehemkay/nixos-fossil`. agenix-encrypted secrets are inert without each host's SSH private key, so public is safe.

```
flake.nix
flake.lock
secrets/
  secrets.nix              # agenix recipient map
  cloudflare-dns.age       # Cloudflare DNS-01 API token
  fossil-sync.age          # sync user password (used by secondaries)
  tmk-password.age         # breaking-glass console password for tmk
  tailscale-authkey-c.age  # Tailscale auth keys, one per host
  tailscale-authkey-s1.age
  tailscale-authkey-s2.age
  healthchecks-s1.age      # healthchecks.io ping URLs, one per monitored timer
  healthchecks-s2.age
modules/
  common.nix               # base: users, ssh, tailscale, firewall, sysctl,
                           # autoUpgrade, journal capping, agenix import
  fossil-server.nix        # the meat: services.fossilServer.* options
  disko.nix                # disk layout, parameterized by device path
hosts/
  canonical.nix
  canonical-hardware.nix   # generated by nixos-generate-config at deploy time
  secondary-1.nix
  secondary-1-hardware.nix
  secondary-2.nix          # added later
  secondary-2-hardware.nix
bin/
  new-repo.sh              # helper: create a repo across the cluster
  smoke-test.sh            # post-deploy verification
README.org                 # quick-start + index
docs/                      # see §7
```

### Module interface (sketch)

```nix
options.services.fossilServer = {
  enable = mkEnableOption "Fossil server";
  role = mkOption { type = enum [ "canonical" "secondary" ]; };
  domain = mkOption { type = str; };                       # e.g. "fossil.exidia.com"
  canonicalUrl = mkOption {                                # only used when role = secondary
    type = nullOr str;
    default = null;
  };
  repoDir = mkOption { type = path; default = "/var/lib/fossil/museum"; };
  syncCredentialFile = mkOption { type = path; };          # agenix-decrypted path
  syncInterval = mkOption {                                # systemd OnCalendar
    type = str;
    default = "*:0/5";
  };
  healthcheckUrlFile = mkOption {                          # null = no monitoring
    type = nullOr path;
    default = null;
  };
};
```

The module:

- Creates the `fossil` system user and group, owns the `repoDir`.
- Configures `security.acme.certs.<domain>` with Cloudflare DNS-01.
- Defines `fossil-server.service` running fossil on port 443 with native TLS.
- If `role == "secondary"`: defines `fossil-sync.service` + `fossil-sync.timer`.
- Sets `reloadServices = [ "fossil-server.service" ]` on the acme cert so it restarts on rotation.

### Per-host files

Each `hosts/<name>.nix` is tiny — imports the common modules, sets the role, domain, and which agenix secrets it can decrypt. Hardware config is generated per-host at deploy time.

### Flake outputs

`flake.nix` exposes two outputs per host — a regular one and a bootstrap one — to resolve the chicken-egg where the first install activates *before* the host's SSH host pubkey has been added to the agenix recipient list (so no secrets are decryptable yet):

- `nixosConfigurations.canonical` — full config (fossil-server, Tailscale auto-up, tmk console password — all reference agenix secrets)
- `nixosConfigurations.canonical-bootstrap` — minimal config: same base (SSH key-only with tmk's pubkeys, sysctl hardening, firewall), but with `services.fossilServer.enable = false`, no `services.tailscale.authKeyFile`, no `users.users.tmk.hashedPasswordFile`, no `security.acme` cert request. References *zero* agenix secrets. Used only for the first `nixos-anywhere` activation.
- Same pair for `secondary-1`, `secondary-2`.

After the bootstrap deploy succeeds and the host's pubkey is rekeyed into the relevant secrets, `nixos-rebuild switch --flake .#<host>` flips everything on.

The bootstrap output is a thin wrapper: `import ./hosts/<host>.nix { bootstrap = true; }`. The host file branches on the flag to disable the secret-dependent options. Both outputs share the same hardware-config import.

## 4. Data Flow & Lifecycle

### TLS cert lifecycle

```
NixOS activation
  └─► security.acme.certs."<domain>"
        ├─► reads cloudflare-dns.age (Cloudflare API token, agenix-decrypted)
        ├─► requests cert via Let's Encrypt DNS-01 challenge
        └─► writes /var/lib/acme/<domain>/{fullchain,key}.pem
              owner: acme group + fossil group (readable by fossil service)

Renewal: systemd timer (acme-<domain>.timer) fires daily.
         Renews if < 30 days remain. On success, triggers
         reloadServices = [ "fossil-server.service" ] which restarts
         fossil (~1s downtime).
```

**Security property (narrow)**: after fossil chroots and drops privileges, it can no longer open the ACME key file from disk. The chroot's filesystem view doesn't include `/var/lib/acme/`, and the dropped-privilege user has no read access to it on the host filesystem. This is an *on-disk attack-surface reduction*, not absolute key secrecy: the TLS key material remains resident in fossil's memory while the process is running, so a memory-disclosure compromise within fossil itself could still expose it. The claim we make is file-access reduction, nothing stronger.

### Fossil request handling

```
fossil-server.service (systemd, User=root)
  └─► fossil server
        --port 443
        --cert /var/lib/acme/<domain>/fullchain.pem
        --pkey /var/lib/acme/<domain>/key.pem
        --repolist /var/lib/fossil/museum
        --baseurl https://<domain>/
        --jsmode bundled
      sequence inside fossil:
        1. bind :443 (needs root)
        2. read --cert and --pkey
        3. chroot into the repolist directory
        4. setuid to repolist owner (the `fossil` user)
        5. accept connections, serve TLS
```

No `--acme` flag (DNS-01 means fossil never participates in the cert challenge). No `--nossl`.

systemd unit hardening:

- `User=root` (needed for the bind+chroot+privdrop sequence)
- `Restart=always`, `RestartSec=3`
- `NoNewPrivileges=true` (blocks setuid-up; allows setuid-down)
- `ProtectHome=true`, `ProtectSystem=strict` with `ReadWritePaths=[/var/lib/fossil]`
- `PrivateTmp=true`
- `ProtectKernelTunables=true`, `ProtectKernelModules=true`, `ProtectControlGroups=true`
- `RestrictNamespaces=true`

### Replication (secondaries only)

```
fossil-sync.timer (systemd, OnCalendar=*:0/5)
  └─► fossil-sync.service (User=fossil, Type=oneshot)
        ├─► fossil all sync -u
        └─► ExecStartPost: curl $(cat $healthcheckUrlFile)
              # pings healthchecks.io after successful sync;
              # hc.io alerts if pings stop arriving
```

**How the sync credential gets there**: `fossil-sync.age` holds the password. Each repo's stored remote URL is set once (during repo provisioning) to `https://syncuser:PASS@fossil.exidia.com/<reponame>` via `fossil remote-url`. The password lives in the repo's SQLite file, readable only by the `fossil` user (file perms `0600`).

**Rotation implication**: the password is embedded in every secondary repo's stored remote URL, not just `fossil-sync.age`. Rotating the credential requires *both* (a) re-encrypting `fossil-sync.age` with the new value and (b) re-running `fossil remote-url -R <repo> https://syncuser:NEW@...` for every repo on every secondary. The rotate-secrets runbook (§7) enumerates this explicitly.

**Conflict semantics**:

- Concurrent commits on different nodes create independent branch leaves; both exist, no destruction. Merge later if needed.
- Wiki/forum/ticket updates: last-writer-wins per field (fossil's documented behaviour).
- Eventual consistency. Worst-case convergence: ~5 min canonical↔secondary (one tick), ~10 min secondary↔secondary via canonical (two ticks). See §2 for the per-direction breakdown.

**Canonical down**:

- Secondaries keep serving local reads/writes.
- Sync timer ticks fail (logged in journal; healthchecks.io alerts after grace period).
- When canonical returns, next tick syncs accumulated changes.
- No automatic failover. Promotion is a manual runbook.

### New-repo provisioning

Fossil's `all` list is per-user, not auto-discovered. Fossil users are *per-repository* (stored in each repo's SQLite), so a freshly `init`'d repo has no `syncuser` even though the cluster-wide password is in `fossil-sync.age`. The provisioning flow must create the per-repo sync user on canonical *before* any secondary clones the repo. The `bin/new-repo.sh <name>` helper performs the full flow across the cluster via SSH:

```bash
# On canonical:
sudo -u fossil fossil init /var/lib/fossil/museum/<name>.fossil
# Create the sync user inside the new repo and grant sync capabilities.
# Capability string "v" = Developer macro (Check-in, Check-out, Clone,
# Hyperlinks, Read/Write ticket, etc.) — covers everything fossil sync needs.
# Verify the exact capability bits against fossil's `user capabilities` docs
# during implementation; tighten if "v" turns out to grant more than required.
sudo -u fossil fossil user new syncuser -R /var/lib/fossil/museum/<name>.fossil
sudo -u fossil fossil user password syncuser "$PASS" -R /var/lib/fossil/museum/<name>.fossil
sudo -u fossil fossil user capabilities syncuser v -R /var/lib/fossil/museum/<name>.fossil
sudo -u fossil fossil all add /var/lib/fossil/museum/<name>.fossil

# On each secondary (only after canonical's syncuser is in place):
sudo -u fossil fossil clone https://syncuser:$PASS@fossil.exidia.com/<name> \
  /var/lib/fossil/museum/<name>.fossil
sudo -u fossil fossil remote-url -R /var/lib/fossil/museum/<name>.fossil \
  https://syncuser:$PASS@fossil.exidia.com/<name>
sudo -u fossil fossil all add /var/lib/fossil/museum/<name>.fossil
```

The password is read from the agenix-decrypted file on each host during the SSH command. The script itself never sees the plaintext.

**Verification**: before exiting, `bin/new-repo.sh` runs `sudo -u fossil fossil all sync -u` on each secondary and asserts a successful sync against the new repo. A failure here typically means the canonical-side `syncuser` step was skipped or the capability string is too narrow.

## 5. Hardening & Operations

### Network surface

| Port  | Proto | Direction | Notes                                                    |
| ----- | ----- | --------- | -------------------------------------------------------- |
| 22    | TCP   | inbound   | SSH, key-only, public + tailscale0                       |
| 443   | TCP   | inbound   | fossil server                                            |
| 41641 | UDP   | inbound   | Tailscale direct-connect (auto-opened by NixOS)          |
| *     | *     | outbound  | default-allow (stateful)                                 |

No port 80 (DNS-01). No cleartext fossil port.

### SSH config

```nix
services.openssh = {
  enable = true;
  settings = {
    PermitRootLogin = "prohibit-password";
    PasswordAuthentication = false;
    KbdInteractiveAuthentication = false;
  };
};
```

### Users

- **root** — no password set. Login impossible via SSH (prohibit-password) or console (no password).
- **tmk** — `isNormalUser`, member of `wheel`, SSH key auth. `hashedPasswordFile = config.age.secrets.tmk-password.path` for breaking-glass console login only (SSH password auth is off).
- **fossil** — system user, no shell, no login, no password. Owns `/var/lib/fossil/`.

```nix
security.sudo.wheelNeedsPassword = false;
users.mutableUsers = false;
```

### Tailscale

```nix
services.tailscale = {
  enable = true;
  authKeyFile = config.age.secrets."tailscale-authkey-${role-suffix}".path;
};
```

NixOS auto-opens UDP 41641. Tailscale auto-connects on first boot using the agenix-decrypted authkey.

### sysctl hardening

Carry forward from prior attempt (`~/dev/playground/nixos-fossil/configuration.nix:241-274`):

- Disable magic SysRq.
- Ignore ICMP broadcasts and bogus error responses.
- RP filter for spoof protection on all interfaces.
- SYN cookies.
- Refuse ICMP redirects (in/out).
- Refuse source-route packets (v4 + v6).
- TCP RFC1337 (time-wait assassination).
- TCP Fast Open mode 3.
- `tcp_congestion_control = bbr`, `default_qdisc = cake`.

Zero maintenance cost, real defensive value.

### Auto-upgrade

```nix
system.autoUpgrade = {
  enable = true;
  flake = "github:teehemkay/nixos-fossil#${hostname}";
  allowReboot = true;
  dates = "Sun 03:00 UTC";
  randomizedDelaySec = "30min";  # stagger reboots
};

nix.gc = {
  automatic = true;
  dates = "Sunday 02:00 UTC";
  options = "--delete-older-than 14d";
};
```

NixOS atomic upgrade + auto-rollback semantics mean a bad upgrade reverts on next boot if activation fails. Staggered reboots ensure not all hosts reboot simultaneously.

### Journal capping

```nix
services.journald.extraConfig = ''
  SystemMaxUse=500M
  MaxRetentionSec=30day
'';
```

### Monitoring (v1)

Only the sync timer is monitored, via healthchecks.io:

- Each secondary has a hc.io check UUID stored in agenix as `healthchecks-<hostname>.age`.
- `fossil-sync.service` `ExecStartPost` curls the ping URL on success.
- hc.io alerts (configured email / Slack / etc.) when pings stop arriving within the grace period.

Other failures: noticed on next SSH + `systemctl --failed`. Document the gap explicitly.

## 6. Deploy Workflow

### Initial deploy (per host)

1. On laptop: `cd ~/dev/my/nixos-fossil`.
2. Provision Hetzner/DO VM; capture public IP.
3. Add Cloudflare A record for the host's DNS name.
4. From laptop, run the bootstrap install (no secrets referenced — see §3 "Flake outputs"):
   `nixos-anywhere --flake .#<hostname>-bootstrap --target-host root@<ip> --generate-hardware-config nixos-generate-config ./hosts/<hostname>-hardware.nix`.
5. After install + reboot, capture the host's new SSH host pubkey (from `/etc/ssh/ssh_host_ed25519_key.pub` via the SSH session — your SSH user pubkeys are already in the bootstrap config so login works).
6. Update `secrets/secrets.nix` to add the host's pubkey to the recipient lists for the secrets this host needs (cloudflare-dns, fossil-sync, tmk-password, tailscale-authkey-<host>, healthchecks-<host> if a secondary).
7. Re-encrypt secrets: `agenix --rekey`.
8. Promote to full config: `nixos-rebuild switch --flake .#<hostname> --target-host root@<ip>`. This is the first activation that references agenix secrets; all of them must decrypt now. ACME issues the TLS cert during this run. Tailscale auto-joins. fossil-server starts.
9. Verify: `bin/smoke-test.sh <hostname>` (TLS cert valid, HTTPS 200 on /timeline.rss, Tailscale joined).

### Ongoing updates

Two paths:

- **Push-style** (immediate, intentional): `nixos-rebuild switch --flake .#<host> --target-host root@<host>`. Drive the rollout; watch output; choose host order. Use `nixos-rebuild test` first for risky changes (activates without making it default-boot; survives reboot via previous generation).
- **Pull-style** (scheduled patches): `system.autoUpgrade` weekly, Sun 03:00 UTC. Hosts pull from `github:teehemkay/nixos-fossil#<hostname>`. Staggered reboots via `randomizedDelaySec`.

### Repo provisioning

`bin/new-repo.sh <name>` — see §4.

### Testing strategy

- **Pre-deploy**: `nix flake check` evaluates all hosts.
- **Per-target eval**: `nix eval ".#nixosConfigurations.<host>.config.system.build.toplevel.drvPath" --raw` for each host (`-hardware.nix` is generated at deploy time so flake-check can't evaluate hosts before their first deploy).
- **Post-deploy smoke**: `bin/smoke-test.sh <host>` curls `/timeline.rss`, asserts TLS validity + 200 response.
- **No VM-based integration tests** for v1. Revisit in v2 if needed.

## 7. Documentation Deliverables

Comprehensive, detailed, actionable. Organized so future-tmk can find the right depth quickly: top-level orientation in `README.org`, then `docs/` for everything else. All in org-mode.

### Layout

```
README.org                          Quick-start, project overview, index
docs/
  architecture.org                  Topology, decisions, rationale, diagrams,
                                    explicit non-goals
  setup.org                         Initial bootstrap + adding a host + adding a repo
                                    (sub-headings per flow)
  operations.org                    Routine checklist + auto-upgrade + agenix workflow
                                    + healthchecks (day-to-day)
  reference.org                     Module options + secrets catalog + network topology
                                    (lookup material)
  runbooks/                         Emergency-time, each standalone, paste-able
    promote-secondary.org
    recover-via-rescue.org
    rotate-secrets.org              One file, sub-sections per secret type
    debug-cert-failure.org
    debug-sync-failure.org
```

### Document standards

Every document follows this shape:

1. **Purpose** — one paragraph: what this doc is for, when to reach for it.
2. **Prerequisites** — what state the system must be in; what credentials/access; what to read first.
3. **Procedure** — numbered, copy-pasteable commands, expected output snippets. Every step actionable: "run X" / "edit Y" / "verify Z" — never "configure the thing".
4. **Verification** — how to confirm success; what failure looks like.
5. **Rollback / recovery** — what to do if the procedure goes wrong mid-way.
6. **Cross-references** — links to related runbooks or reference docs.

Runbooks especially must answer "I have N minutes and the cluster is on fire — what do I type?" Every command paste-able. Every variable substitution shown explicitly (`<hostname>`, `<repo-name>`).

### Content outlines

**`README.org`** — Project README at repo root.

- One-paragraph project description.
- Cluster overview: 3 hosts, what each does, why.
- Quick links to: `docs/setup.org`, `docs/runbooks/`, `docs/architecture.org`.
- Local dev: `nix flake check`, per-target eval.
- License / contact.

**`docs/architecture.org`** — Why the system looks the way it does.

- Topology diagram (canonical + 2 secondaries, sync arrows).
- Decisions and rationale:
  - Fresh repo vs. extending `nixos-anywhere`.
  - Fossil-native TLS vs. reverse proxy.
  - DNS-01 vs. HTTP-01.
  - Canonical+secondaries vs. peer mesh.
  - agenix vs. sops-nix vs. out-of-band.
  - Public flake on GitHub vs. private.
  - healthchecks.io heartbeat vs. SMTP alerts.
- Explicit non-goals (§1).
- Open questions / decisions deferred.

**`docs/setup.org`** — All setup flows; sub-headings per flow.

- **Initial bootstrap**: agenix admin keypair, Cloudflare account + DNS-01 API token, zone delegation from Hover, healthchecks.io account, Tailscale auth keys, initial secret encryption, push to GitHub.
- **Adding a host**: VM provisioning, DNS A record, `nixos-anywhere` invocation, hardware-config generation, host pubkey capture, agenix rekey, follow-up `nixos-rebuild switch`, smoke test, Tailscale verification, ACME cert verification.
- **Adding a repo**: `bin/new-repo.sh` invocation; under-the-hood explanation including the canonical-side `syncuser` creation (per-repo user with sync capabilities — see §4); post-creation verification on each host (the script's own `fossil all sync -u` assertion + a manual check); bootstrap of the human-facing admin user via web UI after the syncuser+sync infrastructure is verified.

**`docs/operations.org`** — Day-to-day.

- **Routine checklist**: weekly (glance at hc.io dashboard), monthly (review autoUpgrade logs), quarterly (review token/cert expiries), annually (review NixOS version pins).
- **Auto-upgrade**: how the timer/service works; what `allowReboot` triggers; how to verify success; how to rollback; how to disable temporarily; failure modes.
- **agenix workflow**: recipient map model; adding a new secret; rotating an existing secret; adding a new host's key; revoking a host.
- **Healthchecks**: what's monitored; how to add a check; how alerts are routed; tuning grace periods.

**`docs/reference.org`** — Lookup material.

- **Module options**: every `services.fossilServer.*` option — type, default, description, example.
- **Secrets catalog**: every agenix secret — purpose, format, generation steps, who reads it.
- **Network topology**: DNS zones/delegations; firewall rules per host; Tailscale tags; outbound endpoints (Cloudflare API, hc.io, GitHub, Let's Encrypt).

**`docs/runbooks/*.org`** — Each standalone, follows the 6-section standard.

- **promote-secondary**: when canonical is lost; elect new canonical; repoint DNS; disable sync timer on promoted host; optionally repoint remaining secondaries.
- **recover-via-rescue**: Hetzner rescue mode + DigitalOcean equivalent; mount disk; edit config; reboot.
- **rotate-secrets**: cloudflare token / fossil-sync password / tailscale authkey — each as a sub-section. The common procedure is re-encrypt the `.age` file + `nixos-rebuild switch` per host. **The fossil-sync sub-section adds a mandatory extra step**: after redeploy, enumerate repos on each secondary and run `fossil remote-url -R <repo> https://syncuser:NEW@<canonical>/<reponame>` for every repo (the old password is embedded in each repo's SQLite-stored remote URL; just rotating `fossil-sync.age` would silently leave existing repos using the old password). Verify with a manual `sudo -u fossil fossil all sync -u -v` per secondary and confirm healthchecks.io pings resume.
- **debug-cert-failure**: where logs live; common DNS-01 failure causes; manual fallback via `nix run`.
- **debug-sync-failure**: inspect `journalctl -u fossil-sync`; manual `fossil all sync -u -v`; check `fossil remote-url`; fix authentication.

### Documentation lifecycle

- All docs live in the repo, version-controlled.
- Code changes that affect a documented procedure must update the doc in the same commit (enforced by review).
- Each runbook is exercised on a real or rehearsal host at least once before being trusted; corrections folded back.
- Definition-of-done for a feature includes: code + tests + the documentation that documents it.

## 8. Acceptance Criteria

The v1 implementation is complete when:

1. `nix flake check` evaluates cleanly (allowing for `*-hardware.nix` throw-placeholders on hosts not yet provisioned).
2. Per-target eval succeeds for every provisioned host.
3. `nixos-anywhere` bootstraps a fresh host from zero in one invocation (plus the secret-rekey + follow-up rebuild).
4. The canonical and at least one secondary are deployed; sync timer succeeds; a canonical-originated write reaches the secondary within ~5 min (one timer tick). With both secondaries deployed, a secondary-originated write reaches the other secondary within ~10 min (two timer ticks).
5. TLS certs are issued for both deployed hosts; HTTPS to each host returns a valid fossil page.
6. SSH works via both public and Tailscale paths.
7. The breaking-glass console password unlocks tmk's account on the browser console.
8. All documentation in §7 is written and exercised at least once.
9. `bin/smoke-test.sh` and `bin/new-repo.sh` work end-to-end against the deployed cluster.

## 9. Open Questions Deferred to Plan

- Exact systemd unit syntax for fossil-server (capability flags, chroot interaction).
- Whether to use `services.tailscale.authKeyFile` (NixOS option in 24.05+) or a bootstrap unit running `tailscale up --authkey=...`.
- Cloudflare API token scope: zone-specific (`Zone:DNS:Edit` for `fossil.exidia.com`) vs. account-wide. Lean: zone-specific.
- Hetzner CPX size: CPX11 (2 vCPU, 2GB RAM, €4.51/mo) is likely sufficient; confirm during plan.
- DigitalOcean region for secondary-2: pick based on geo-diversity from Hetzner DCs.
- Whether the helper scripts (`bin/new-repo.sh`, `bin/smoke-test.sh`) should be bash or Babashka. Lean: bash (one fewer dependency on each host).
