** Gotchas

- agenix 0.15.0 only auto-discovers =~/.ssh/id_ed25519= and =~/.ssh/id_rsa= as decryption identities (verified in the agenix script, lines 137-142). It does *not* read =~/.ssh/config= =IdentityFile=. Because the new admin key sits at the non-default =~/.ssh/id_ed25519_ext=, every agenix call that *decrypts* (=-e= on a real file, =-d=, =--rekey=) now needs =-i ~/.ssh/id_ed25519_ext=. First encryption after =rm= needs no =-i= — nothing to decrypt.
- The =id_ed25519_ext= private key is passphrase-protected. Decryption can't be verified from the non-interactive Bash tool — age tries to prompt and fails with "standard input is not a terminal, and /dev/tty is not available". Verification had to be handed to tmk's terminal.
- The =agenix --rekey= for the key migration itself needed *no* =-i=: rekey decrypts the existing ciphertext with a *current* recipient's key — the OLD key — and that one happened to live at the default =~/.ssh/id_ed25519=, so agenix auto-found it. Only post-migration operations need the flag.

* Technical Insights

** Re-keying is asymmetric: it decrypts with the old recipient, encrypts to the new

When migrating an age/agenix recipient, =--rekey= must first decrypt every file using a key that is *already* a recipient — i.e. the OLD key. It then re-encrypts to whatever the rules file now lists — the NEW key. So the rekey step's auth requirements differ from every step after it. If the old key sits at a tool-default path and the new one doesn't, the rekey "just works" while all later edits suddenly need an explicit identity flag. Easy to mis-document if you reason about it as one uniform operation.

** A config repo is never just config

The architecture doc had a vetted decision record — "the public flake doesn't expose anything sensitive" — reasoning correctly about encrypted =.age= files and public-by-DNS hostnames. But =git ls-files= showed =journal/=, =todo.org=, and =docs/superpowers/= riding along in the same repo. The doc's rationale was true within its scope and silently wrong outside it. Before acting on "make it public", enumerate what's actually tracked — and remember git history makes the decision permanent.

** Verifying secret decryption without leaking the secret

To confirm a rekeyed =.age= file still decrypts, pipe the plaintext to =wc -c= rather than letting it hit the terminal: a positive byte count proves decryptability, the secret never appears. (Here: 65 bytes = a 64-char =pwgen= password + newline — the count itself corroborated the content.)
