* Technical Insights

** Two-tier flake outputs for nixos-anywhere repos

A NixOS flake that uses =nixos-anywhere= needs hardware-config to exist *somewhere*, but provisioned hosts haven't generated it yet on first install. The plan's two-tier output pattern:

- =hosts/<name>-hardware.nix= is a =throw ''...''= placeholder until =nixos-anywhere --generate-hardware-config= overwrites it during install.
- =flake.nix= exposes =<host>= (full config, throws on hardware) AND =<host>-bootstrap= (no agenix, no fossil-server — just =common.nix= + =disko.nix=). Bootstrap is what =nixos-anywhere= activates first.

But the throw masks any other wiring errors below it. Solution: a third tier, =<host>-eval-test=, that swaps in a tiny non-throwing =_fixture-hardware.nix= fixture. Lets you verify the entire module composition evaluates without doing a real hardware scan. This is the load-bearing pre-deploy verification.

** mkIf inside attrsOf submodules strips the attribute cleanly

The =fossil-sync.service= definition needs an =ExecStartPost= that only fires when =healthcheckUrlFile= is non-null. Plan does:

#+begin_src nix
ExecStartPost = lib.mkIf (cfg.healthcheckUrlFile != null) (
  pkgs.writeShellScript "fossil-sync-healthcheck" '' ... ''
);
#+end_src

Reviewer initially wondered if =mkIf false= at the attribute-value level would produce a partial unit. Confirmed: within a typed =attrsOf= submodule, =mkIf false= is processed by the module system and the attribute is simply absent from the final attrset. Safe pattern.

** Single-quoted heredocs are the safe SSH-with-args idiom

=bin/new-repo.sh='s pattern:

#+begin_src bash
remote "$CANONICAL" bash -s -- "$REPO" <<'EOF'
set -euo pipefail
REPO="$1"
# ... uses $REPO safely
EOF
#+end_src

The =<<'EOF'= (single-quoted) prevents local expansion of =$REPO=, =$PASS= etc. inside the heredoc body. Variables reach the remote only through the =bash -s -- arg1 arg2= argv mechanism, where they become =$1=, =$2=. Combined with input validation against a tight alphanumeric regex, shell injection via the repo name argument is structurally impossible. Pair with =set -euo pipefail= *inside* the heredoc to propagate remote errors back through the SSH exit code.
